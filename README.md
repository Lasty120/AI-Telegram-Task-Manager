# AI Telegram Task Manager

Telegram-бот — умный менеджер задач, разработанный для решения проблем офиса Almaty Hub в рамках производственной практики. 

Приложение понимает естественную речь (текст и голос), сам превращает её в структурированные задачи через LLM и синхронизирует их с **Notion**. Никаких команд с жёстким синтаксисом: просто напишите или надиктуйте, что нужно сделать, а бот разберётся с датой, приоритетом, длительностью и деталями сам.

## Возможности

### Понимание естественного языка

Бот не требует специального синтаксиса команд. Любое сообщение (текстовое или голосовое) отправляется в LLM с детальным системным промптом, который извлекает из него одну или несколько задач и определяет нужное действие:

* **Создание** задач — «напомни позвонить маме завтра в 18:00», «встреча с командой в пятницу в 10:00, высокий приоритет»
* **Изменение** — «перенеси встречу с командой на 12:00», «измени приоритет задачи про отчёт на высокий»
* **Завершение** — «я купил хлеб, отметь задачу выполненной»
* **Удаление** — «удали задачу про звонок маме»
* **Поиск** — «что у меня запланировано на завтра?», «покажи все срочные задачи»
* **Работа с Notion прямо из чата** — «создай задачу и добавь в ноушн», «переведи задачу 3 в статус в работе», «добавь комментарий к встрече: взять ноутбук»

Для каждой задачи ИИ автоматически (если не указано явно) определяет:

* **дату и время** — рассчитывается относительно текущего момента с учётом часового пояса;
* **длительность** — логическая оценка на основе сути задачи (встреча ≈ 60 мин, быстрое дело ≈ 5 мин и т.д.);
* **приоритет** (`low` / `medium` / `high`) — только если пользователь явно об этом упомянул;
* **детали** — перевод слова, контекст события, список пунктов и т.п.

### 🎙 Голосовые сообщения

Голосовые сообщения скачиваются с серверов Telegram и транскрибируются через **Groq Whisper** (`whisper-large-v3-turbo`), после чего обрабатываются тем же пайплайном, что и обычный текст.

### 📅 Умное разрешение конфликтов расписания

Если новая задача пересекается по времени с уже существующей, бот:

1. автоматически ищет ближайший свободный слот нужной длительности, если время не было указано пользователем явно;
2. если время указано явно и конфликтует с другой задачей — присылает интерактивное предупреждение с кнопками: *оставить обе задачи параллельно*, *сдвинуть старую*, *сдвинуть новую*.

### 🔗 Двусторонняя интеграция с Notion

* **Экспорт**: созданные в боте задачи можно одним запросом отправить в Notion — поля (заголовок, дата, описание, приоритет, статус, мультиселект/спринт, автор) автоматически сопоставляются со свойствами вашей базы данных по их типу.
* **Импорт** (`/fetch\_notion\_tasks`): подтягивает из Notion все незавершённые задачи, назначенные на пользователя, и заводит их локально — с них тоже будут приходить напоминания.
* **Синхронизация статусов**: кнопка «Синхронизировать выполненные задачи из Notion» помечает в боте выполненными все задачи, чей статус в Notion начинается на `Done`.
* **Комментарии**: можно попросить бота оставить комментарий к конкретной задаче прямо в Notion.
* Настройка отдельного статуса «при создании» и «при выполнении» задачи, поддержка нескольких источников данных (data sources) внутри одной базы.

### 🛡 Одобрение администратором

Привязка Notion-аккаунта пользователя проходит модерацию: после выбора участника рабочего пространства Notion всем администраторам (`ADMIN\_IDS`) приходит запрос с кнопками «Одобрить» / «Отклонить», и только после подтверждения интеграция активируется.

### 🔔 Напоминания и рассылки

* Индивидуальное уведомление точно в момент наступления задачи (с кнопками «Выполнить» / «Отложить на 15 минут» прямо в уведомлении).
* Ежедневные сводки задач в **9:00** и **21:00**.
* При перезапуске бота все активные задачи заново подгружаются в планировщик из базы данных — напоминания не теряются.

### 🗂 Гигиена базы задач

Отдельный раздел «Просроченные задачи» показывает все задачи с истёкшим сроком и позволяет массово завершить их или почистить синхронизацией с Notion — это помогает не допускать разрастания базы (при 2000+ задачах в контексте ИИ начинает путаться).

### 🌍 Мультиязычность

Русский и английский интерфейс, переключается командой `/change\_language`.

### 📄 Пагинация

Списки задач (активные, выполненные, на сегодня, результаты поиска) отображаются постранично с инлайн-навигацией.

## 🏗 Архитектура

Проект построен на **aiogram 3.x** с чистым разделением на слои и Dependency Injection: репозитории создаются в хендлерах и передаются в сервисы, сервисы не знают о существовании друг друга напрямую, кроме как через явно переданные зависимости.

```
┌─────────────┐     ┌──────────────┐     ┌───────────────┐
│  handlers/  │ ──▶ │  services/   │ ──▶ │ repositories/ │ ──▶ PostgreSQL
│ (aiogram)   │     │ (бизнес-лог.)│     │  (asyncpg)    │
└─────────────┘     └──────────────┘     └───────────────┘
       │                    │
       │                    ├──▶ services/ai      (LLM: разбор команд)
       │                    ├──▶ services/notion   (Notion API)
       │                    └──▶ services/scheduler (APScheduler)
       │
       └──▶ middlewares/ (сессия БД на Update + получение/создание пользователя)
```

### Ключевые слои

|Слой|Назначение|
|-|-|
|`handlers/`|Точки входа aiogram: команды, текстовые кнопки, callback-запросы. Ничего не знают про SQL — только вызывают репозитории/сервисы.|
|`services/ai/`|Формирование системного промпта и обращение к LLM через OpenAI-совместимый клиент; возвращает провалидированный Pydantic-объект `MultiTaskActionSchema`.|
|`services/notion/`|Клиент Notion API, автомаппинг полей задачи на свойства БД по их типу, импорт/экспорт, работа со статусами и комментариями.|
|`services/tasks/`|CRUD над задачами, разрешение конфликтов расписания, синхронизация с планировщиком и Notion, точка входа `processor.py`, которая раздаёт разобранные ИИ-команды по нужным сервисам.|
|`services/scheduler.py`|Глобальный инстанс `APScheduler`, разовые и cron-джобы уведомлений.|
|`database/repositories/`|Единственный слой прямого доступа к PostgreSQL (`asyncpg`).|
|`database/migrations/`|Версионируемые `.sql`-миграции с собственным раннером (без Alembic).|
|`middlewares/`|Выдаёт каждому Update изолированное соединение из пула в рамках транзакции и подготавливает объект `user`.|
|`keyboards/`|Reply- и inline-клавиатуры.|
|`messages.py`|Все пользовательские тексты бота на двух языках в одном месте.|
|`utils/`|Форматирование списков задач, пагинация, работа с датами и таймзонами.|

### Про базу данных

Проект **мигрировал с SQLite (`aiosqlite`) на PostgreSQL (`asyncpg`)**. Файл `database/init\_db.py` — устаревший, оставлен для истории и не используется в актуальном пайплайне запуска; реальная схема применяется через `database/migrations/\*.sql` и раннер `database/migrations/runner.py`, который сам отслеживает уже применённые миграции в служебной таблице `schema\_migrations`. При добавлении новой миграции достаточно положить `.sql`-файл с очередным числовым префиксом (`003\_...`) — он подхватится и применится автоматически при следующем запуске.

### Про LLM-клиента

`services/ai/service.py` использует `AsyncOpenAI`-клиент, поэтому провайдера можно менять через `.env` (`OPENAI\_DEFAULT\_URL`, `OPENAI\_DEFAULT\_MODEL`) — подходит любой OpenAI-совместимый эндпоинт: сам OpenAI, Gemini через совместимый прокси, xAI, локальные модели (LM Studio, Ollama с OpenAI-совместимым сервером) и т.д. Для транскрипции голоса используется отдельный ключ **Groq**.

## 🛠 Технологический стек

* **Python 3.12**
* [**aiogram 3**](https://docs.aiogram.dev/) — асинхронный фреймворк для Telegram Bot API, FSM для многошаговых сценариев (регистрация Notion)
* **PostgreSQL +** [**asyncpg**](https://github.com/MagicStack/asyncpg) — основное хранилище, без ORM, с ручным пулом соединений
* [**APScheduler**](https://apscheduler.readthedocs.io/) — планировщик разовых напоминаний и ежедневных рассылок
* [**OpenAI Python SDK**](https://github.com/openai/openai-python) — унифицированный клиент для любого OpenAI-совместимого LLM-провайдера, структурированный вывод в JSON-режиме
* [**Groq**](https://groq.com/) **(Whisper Large v3 Turbo)** — распознавание речи
* **Notion API** (версия `2025-09-03`, с поддержкой `data\_sources`) — через собственный тонкий асинхронный клиент на `aiohttp`
* **Pydantic** — валидация структурированного ответа ИИ
* **aiohttp** — health-check HTTP-сервер и HTTP-клиент для Notion
* **Docker** — готовый `Dockerfile` для продакшен-развёртывания

## 🚀 Установка и запуск

### Предварительные требования

* Python 3.12+
* Запущенный сервер PostgreSQL
* Токен Telegram-бота от [@BotFather](https://t.me/BotFather)
* API-ключ [Groq](https://console.groq.com/) (для распознавания голоса)
* API-ключ LLM-провайдера, совместимого с OpenAI (OpenAI, Gemini, xAI, локальный сервер и т.п.)
* *(опционально)* интеграция Notion — токен создаётся пользователями бота самостоятельно через `/add\_notion`, глобально в `.env` настраивать её не обязательно

### 1\. Клонирование репозитория

```bash
git clone <URL\_ВАШЕГО\_РЕПОЗИТОРИЯ>
cd <ИМЯ\_ПАПКИ\_ПРОЕКТА>
```

### 2\. Переменные окружения

Скопируйте `.env.example` в `.env` и заполните своими значениями:

```bash
cp .env.example .env
```

```env
TG\_BOT\_TOKEN=your\_telegram\_bot\_token\_here
GROQ\_API\_KEY=your\_groq\_api\_key\_here

# OpenAI API (OpenAI, Gemini, xAI, Local, etc.)
OPENAI\_API\_KEY=your\_llm\_api\_key\_here
OPENAI\_DEFAULT\_URL=https://api.openai.com/v1
OPENAI\_DEFAULT\_MODEL=gpt-4o-mini

# Формат: Continent/City (America/New\_York)
TIMEZONE=your\_timezone\_here

# id1,id2,id3,...
ADMIN\_IDS=list\_of\_tg\_ids\_of\_admins

# PostgreSQL
DB\_HOST=
DB\_PORT=
DB\_USER=
DB\_PASS=
DB\_NAME=
```

> \*\*Про `ADMIN\_IDS`\*\*: это список Telegram ID администраторов через запятую (без пробелов или с ними — обрезаются автоматически). Именно этим людям будут приходить запросы на одобрение привязки Notion-аккаунтов.

### 3a. Запуск локально

```bash
python -m venv venv
source venv/bin/activate  # Windows: venv\\Scripts\\activate

pip install -r requirements.txt

python main.py
```

При старте бот сам применит недостающие SQL-миграции к указанной базе PostgreSQL — создавать таблицы вручную не нужно.

### 3b. Запуск через Docker

```bash
docker build -t task-manager-bot .
docker run --env-file .env -p 80:80 task-manager-bot
```

Внутри контейнера параллельно с long polling поднимается лёгкий `aiohttp`-сервер на порту `80` с эндпоинтом `/healthz` — он не участвует в логике бота и нужен исключительно для прохождения health-check проверок платформ развёртывания (Kubernetes, PaaS и т.п.).

## 📋 Команды бота

|Команда / кнопка|Описание|
|-|-|
|`/start`|Приветствие и краткая инструкция по возможностям бота|
|`/add\_notion`|Пошаговая FSM-настройка интеграции с Notion (токен → база данных → источник данных → участник → статусы)|
|`/fetch\_notion\_tasks`|Импортирует незавершённые задачи, назначенные на пользователя, из Notion в бота|
|`/change\_language`|Переключает интерфейс между русским и английским|
|`/cancel`|Отменяет текущий шаг настройки Notion|
|«Мои задачи»|Активные задачи с пагинацией|
|«Мои выполненные задачи»|Завершённые задачи с пагинацией|
|«Задачи на сегодня»|Активные задачи с дедлайном на сегодня|
|«Просроченные задачи»|Задачи с истёкшим сроком + массовые действия и синхронизация с Notion|
|*любой текст или голосовое*|Обрабатывается ИИ как одна или несколько команд над задачами|

## ⚠️ Важные заметки по эксплуатации

* **Гигиена базы задач.** При накоплении более \~2000 задач у одного пользователя ИИ, получающий их список как контекст, начинает ошибаться при сопоставлении команд с конкретными задачами. Рекомендуется периодически завершать/удалять старые задачи — для этого в боте предусмотрен раздел «Просроченные задачи».
* **Одобрение администратором обязательно.** Пока хотя бы один пользователь из `ADMIN\_IDS` не нажмёт «Одобрить», привязка аккаунта Notion остаётся в статусе ожидания и не активируется.
* **Задачи без срока.** Задачам, у которых явно не указан дедлайн, присваивается технический timestamp-заглушка (01.01.2060) — они не попадают в напоминания и отображаются в списках отдельным блоком «Без даты».
* **`init\_db.py` — легаси.** Не используйте этот файл как источник правды о схеме БД: актуальная схема живёт в `database/migrations/`.

## 🔐 Безопасность

* Файл `.env` и любые файлы с реальными базами данных (`\*.db\*`) исключены из git через `.gitignore` — никогда не коммитьте секреты.
* Все токены (Telegram, Groq, LLM-провайдер, Notion) читаются исключительно из переменных окружения.
* Соединение с БД на каждый Update оборачивается в транзакцию — при необработанном исключении в хендлере изменения автоматически откатываются, что защищает от частично применённых операций.

> 💡 Если в процессе разработки вы когда-либо коммитили или пересылали файлы вида `test.py`/`script.py`/`result.txt` с реальными ключами или токенами внутри — обязательно отзовите (revoke) эти ключи в соответствующих сервисах и сгенерируйте новые, даже если сами файлы сейчас в `.gitignore`.
